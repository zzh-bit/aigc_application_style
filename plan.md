# PS²（Parallel Self 2.0）18 天完善开发计划（按功能清单重排）

> 目标：在 18 天内完成可演示、可构建、可打包 Android APK 的完整版本，覆盖 `function.md` 的核心能力：多自我决策、导师智库、记忆融合、情绪感知、未来信件、可视化推演、成长洞察、深度定制。
>
> 当前基础：你已具备 Next.js 前端原型、核心页面骨架、部分 API（chat/emotion/summarize）与 Council 归档能力。

---

## 一、交付定义（18 天后必须达到）

### 1) Web 交付
- `npm run dev`、`npm run lint`、`npm run build` 全通过。
- 7 个核心模块都可演示：Council / Mentor / Memory / Letters / Projection / Breathing / Insights。
- 至少 4 条 AI 链路可用：议会辩论、导师聊天、记忆摘要、洞察总结。

### 2) Android 交付
- Android Studio 工程可运行，APK 可安装。
- 离线可打开 UI，联网可调用 AI。
- 至少 1 个原生增强功能可演示（分享/导出/通知三选一）。

### 3) 数据与安全
- 关键用户数据可持久化（IndexedDB/localStorage + 后端归档）。
- API Key 仅在服务端，前端与 APK 不泄露密钥。
- 基础限流与错误兜底可用。

## 三、18 天逐日详细计划

### Day 1：功能冻结 + 数据模型对齐
- 目标：把 `function.md` 变成可开发任务清单，避免后续返工。
- 任务：

  - 梳理 API 契约：输入输出字段统一（summary/emotions/keywords/date）。
- 验收：
  - 形成一张“模块 -> 页面 -> API -> 存储”的映射表并写入文档。

### Day 2：Council 决策链路固化（你已完成部分，做收口）
- 目标：形成“输入议题 -> 3 派系 -> 主持人总结 -> 决策完成归档”的稳定闭环。
- 任务：
  - 校准四角色 prompt（激进/保守/未来/主持人）。
  - 强化“口语化 + 简洁（<=3句）”输出约束。
  - 完成“每轮决策按钮只可点击一次 + 本页平滑重置”体验细节。
- 验收：
  - 连续 5 轮对话可稳定运行，无状态错乱。

### Day 3：记忆锚点基础 RAG（本地检索版）
- 目标：角色回复能稳定参考历史记忆，而不是随机引用。
- 任务：
  - 新增 memory 检索器：按关键词+时间+类型做 Top-K 检索。
  - Council/mentor 请求统一携带 memory context。
  - 提供“本轮引用了哪些记忆”的可见提示（简短标签）。
- 验收：
  - 用户更改记忆内容后，下一轮回复能体现变化。

### Day 4：决策归档标准化 + 后端查询完善
- 目标：归档数据可用于后续洞察计算。
- 任务：
  - 统一归档结构（summary<=50词、emotion数组、keywords、date、messageCount）。
  - 新增查询参数：`limit`、`from`、`to`、`emotion`。
  - 增加错误码与空数据兜底返回。
- 验收：
  - 在后端可查到过滤后的历史决策列表。

### Day 5：导师智库升级（角色风格差异化）
- 目标：导师不只是“换名字”，而是有明显思维模型差异。
- 任务：

  - 输出中增加“建议卡片”字段（bullet points）。
- 验收：
  - 任意 2 位导师对同问题的回答风格明显不同。

### Day 6：Mentor Chat 流式质量优化
- 目标：提升聊天可用性与稳定性。
- 任务：
  - 完善 SSE 断流重试与取消状态提示。
  - 增加“生成中/已取消/重试”统一 UI 状态。
  - 避免重复拼接、空 token 问题。
- 验收：
  - 10 次连续提问无卡死、无重复段落。

### Day 7：情绪感知闭环（文本版主线，声纹版预留）

  - 为“声纹识别”预留接口与开关（本期可用 mock）。
- 验收：

### Day 8：叙事推演与决策树 API

  - 增加“分支比较视图”（至少2条路径对比）。
- 验收：

### Day 9：Memory Vault 深化（摘要、标签、时间线）

  - 新增异步摘要任务（保存后后台生成 summary）。

- 验收：
  - 新增记忆后可自动完成摘要并支持快速检索。

### Day 10：未来信件主闭环（时空对话系统）
- 目标：完成写信/收信/回信完整流程。
- 任务：
  - 信件模型：triggerType（日期/情绪/事件）、deliveryDate、status、threadId。

### Day 11：智能触发策略（日期+情绪+事件）

## 1. 总体架构（推荐：最稳、最容易按期交付）

### 1.1 运行形态
- 开发阶段：Web 本地运行（Next dev），AI 由本地或云端后端代理提供。
- 演示阶段（APK）：Android WebView 加载“离线静态资源”（最稳，不依赖网络）；AI 功能通过网络调用后端（若无网则降级为本地 mock/提示）。

### 1.2 必须遵守的安全规则
- AI Provider 的 Key 永远不出现在前端代码/Android APK 内。
- 前端只请求你自己的后端（BFF），后端再去请求 OpenAI/DeepSeek/通义等模型。
- 后端要加：限流（简单即可）、基础鉴权（演示期可用“临时 token + 环境变量”）。

### 1.3 模块优先级（按风险从高到低）
1) AI 对话链路（流式/并发/取消）——最容易踩坑，先打通  
2) 数据存储（记忆库/信件）——容易做一半发现不好保存  
3) Android 打包（离线资源 + WebViewAssetLoader）——演示稳定性关键  
4) 动画与 UI 细节——最后再抛光

## 2. 20 天排期总览（每天都有可验证产出）

> 每天结束前必须做到：能运行、能演示、能截图/录屏、能写下“今天完成了什么 + 明天做什么”。

### Day 1：环境与基线
- 目标：电脑环境一次性配好，项目可重复运行。
- 任务：
  - 安装：Node.js LTS、Android Studio、JDK 17、Git。
  - 运行 Web：`npm install`、`npm run dev`、`npm run build`。
  - 记录运行命令与常见报错处理到本文件“附录 A”。
- 验收：
  - 浏览器能打开首页并切换主要入口（Welcome → Council/Mentor/Letters）。

### Day 2：梳理功能清单 → 变成“可交付需求”
- 目标：把 7 大模块拆成“必须有 / 可简化 / 可延后”。
- 任务：
  - 按模块列出 UI 交互点与数据来源（用户输入、AI 输出、本地存储）。
  - 定义演示范围（20 天必须完成的“最小可演示闭环”）：
    - Council：输入议题 → 6 角色输出 → 可继续追问
    - Mentor：选导师 → 聊天 → 可返回
    - Projection：给出 3 派系建议（AI 或 mock）
    - Breathing：能触发弹窗并运行动画
    - Memory：可新增/查看/搜索（AI 摘要可先异步生成）
    - Letters：写信/收信/时间长河展示（触发机制先做“演示版”）
    - Insights：展示统计 + AI 解读（可选）
- 验收：
  - 输出一张“模块-数据-状态”表（写在本文件里即可）。

### Day 3：建立“数据层”与本地存储（先不接 AI）
- 目标：所有模块先跑“本地数据 + mock AI”，避免卡在后端。
- 任务：
  - 选存储：IndexedDB（推荐）或 localStorage（演示可用但容量/结构差）。
  - 定义数据模型（TS 类型）：
    - Conversation/Message（含 role、content、timestamp、emotion?）
    - MemoryItem（type、tags、summary、mood、createdAt）
    - Letter（trigger、deliveryDate、status、content）
    - Decision（topic、options、factions、confidence）
  - 封装 `lib/storage.ts`（读写/迁移/清空/导出 JSON）。
- 验收：
  - Council、Memory、Letters 都能把“用户输入”落到本地并刷新可见。

### Day 4：搭建后端 BFF（最薄一层）
- 目标：前端不直连 AI，先把“前端 → 后端 → 返回”打通（可先返回 mock）。
- 任务（两种选择，二选一即可）：
  - A：用 Next.js Route Handlers（在当前项目内新增 `app/api/*`）
  - B：单独 Node/Express 服务（`server/` 目录，端口 3001）
- 必备接口（先做 mock）：
  - `POST /api/chat`：输入 messages + mode → 返回 assistant 回复
  - `POST /api/summarize`：输入 text → 返回 summary
  - `POST /api/emotion`：输入 text → 返回 emotion label + score
- 验收：
  - 前端能调用 `/api/chat` 并显示返回内容（哪怕是 mock）。

### Day 5：接入第一个真实 AI（普通非流式）
- 目标：完成真实模型调用（最简单版本），验证 Key 安全与网络链路。
- 任务：
  - 选定一个 Provider（OpenAI/DeepSeek/通义/智谱等），把 Key 放在 `.env.local` 或后端环境变量。
  - 后端实现：把 `/api/chat` 从 mock 替换为真实请求（先非流式）。
  - 加基础保护：简单限流（按 IP/按 token）。
- 验收：
  - Council 里输入议题，能返回 AI 文本（先用单一角色/单一回复也行）。

### Day 6：升级为“流式输出”（SSE 优先）
- 目标：让聊天气泡能边生成边显示。
- 任务：
  - 后端：支持 SSE（Server-Sent Events）或 WebSocket。
  - 前端：实现流式拼接渲染（逐 token 更新最后一条消息）。
  - 增加“取消生成”按钮（AbortController）。
- 验收：
  - Mentor Chat 能流式输出并可取消。

### Day 7：Council：3角色并行/轮流生成（演示版可简化）
- 目标：议会辩论形成“3 个平行自我（激进派，未来派，保守派）”的体验。
- 任务：
  - 角色可以根据记忆库里的内容了解用户之前给出情况并有相应回复
  - 角色提示词（system prompt）在后端维护：4 份（还包括主持人）。
  - 生成策略（演示推荐）：顺序生成（更稳）或 2~3 并发（更快但更复杂）。
  - UI：每个角色固定颜色/座位，对应气泡样式。
- 验收：
  - 输入一个议题， 3 个角色能给出不同观点；最后主持人总结。

### Day 8：情绪检测（触发冥想弹窗）
- 目标：实现“高焦虑自动触发 Breathing Guide”闭环。
- 任务：
  - `/api/emotion`：用模型做分类（或规则 + 模型混合）。
  - 前端：当 score 超阈值（例如 > 0.75）弹出 breathing-guide。
  - 允许用户关闭并“本次不再提示”。
- 验收：
  - 输入一段焦虑文本，能触发冥想引导弹窗。

### Day 9：Projection：决策推演（结构化输出）
- 目标：决策树/三派系意见可从 AI 获得结构化结果。
- 任务：
  - `/api/projection`：返回 JSON（options、paths、factions、support）。
  - 前端：渲染你现有决策树组件（若 UI 已完成则接数据；否则先用 mock）。
- 验收：
  - 点击路径 → 显示三派系 → 点击派系查看详细意见与支持度（数据来自接口）。

### Day 10：Memory Vault：AI 摘要与标签
- 目标：记忆卡片支持 AI 摘要（异步生成），并可按年份/类型筛选。
- 任务：
  - 新增记忆：用户输入原文 → 先保存 → 后台调用 `/api/summarize` 生成 summary 并回填。
  - 基础检索：按关键词/类型/年份筛选。
- 验收：
  - 新建一条记忆后，几秒内出现 AI 摘要；重启页面仍存在。

### Day 11：Mentor Library：8 导师人设与对话风格
- 目标：8 位导师可用，风格明显不同。
- 任务：
  - 后端：导师配置表（id、name、category、prompt）。
  - 前端：导师列表 → 点击进入 mentor-chat（带 mentorId）。
  - 增加“导师建议卡片”（从模型输出中提取 bullet）。
- 验收：
  - 任意选择 2 位导师，对同一问题回复风格明显不同。

### Day 12：Future Letters：写信/收信/回信闭环（演示版）
- 目标：完整 UI 与数据闭环先做完，“触发”先做可演示版本。
- 任务：
  - 写信：保存到本地（含 triggerType/triggerValue/deliveryDate）。
  - 收信：按“日期已到/状态”筛选展示。
  - 回信：保存为关联线程。
  - 时间长河：节点点击 → 列表 → 打开信件。
- 演示版触发策略（先这样保证交付）：
  - 日期触发：用本地时间判断是否到期。
  - 情绪触发：当 `/api/emotion` 达阈值时提示“有一封信到达”。
  - 事件触发：用 UI 选择事件后立即触发（演示用）。
- 验收：
  - 写一封“未来信” → 通过按钮模拟触发 → 能在收件箱看到并回信。

### Day 13：Data Insights：统计与 AI 解读
- 目标：洞察页面可展示关键统计，并由 AI 输出解释。
- 任务：
  - 本地统计：决策次数/分类分布、情绪分布、趋势数据（可先 mock 再逐步真实）。
  - `/api/insights`：输入统计摘要 → 输出建议与解读。
- 验收：
  - Insights 页面能稳定展示图表，且能生成一段“本月建议”。

### Day 14：账号/同步（可选但很建议）

### Day 16：静态导出（为离线 APK 做准备）

- 输入议题后有输出（至少 3 角色，最好 6）。
- 输出是流式呈现（或至少有加载态）。

- 点击路径与派系能看到详细内容。

- 动画可运行（吸/停/呼节奏）。

### Memory Vault（记忆库）
- 可新增/浏览/筛选/查看详情。
- AI 摘要可生成并持久化（失败也不影响保存原文）。

### Mentor Library（导师库）

- 至少 2 位导师的风格差异明显。

### Future Letters（未来信件）
- 写信/收信/回信闭环可演示。
- 时间长河可打开并点击节点查看信件列表。
- 触发机制演示版可用（日期/情绪/事件至少各 1 种可触发）。

### Data Insights（数据洞察）
- 至少 1 个决策统计 + 1 个情绪统计图表可见。
- AI 可生成“洞察总结”（失败可降级为本地固定文案）。

## 4. 风险清单与应对（保证 20 天交付）

### 风险 1：AI Key 泄露/费用失控
- 应对：只放后端；加限流；加每日预算阈值（演示期可用固定额度）。

### 风险 2：流式输出做不稳
- 应对：先非流式打通（Day 5）再流式（Day 6）；SSE 优先于 WebSocket。

### 风险 3：Future Letters 的“触发提醒”做不完
- 应对：先做“演示触发按钮 + 情绪触发提示”；真正定时/通知留到 Day 18 做原生增强。

### 风险 4：WebView 兼容性/性能
- 应对：移动端降级粒子/模糊；提供“低性能模式”开关。

### 风险 5：数据丢失
- 应对：本地存储 + 导出 JSON；可选做同步（Day 14）。

## 5. 每日固定流程（零经验也能执行）

- 早上（30~60 分钟）：只做一件事：把今天目标写成 3 条验收标准。
- 白天：只围绕验收标准写代码，不做美化。
- 晚上（30 分钟）：跑一遍：
  - `npm run lint`
  - `npm run build`
  - 打开页面走一遍关键路径
- 记录：今天完成/遇到的问题/明天第一件事。
